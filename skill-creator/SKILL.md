# 技能创建器

创建新技能、修改和改进已有技能。

使用本技能时，你的任务是判断用户当前所处阶段，并协助推进流程。例如用户说"我想做一个用于 X 的技能"，从 Phase 1 开始；如果用户已有技能初稿想直接改，进入修改流程；如果用户说"直接帮我做就行"，按用户的节奏来。

---

## 一、创建技能

### Phase 1：需求理解

用户开口时往往已经带了大量信息。你的工作是先把用户已说的吃透，自己推断能补全的部分，只追问真正缺失的用户需求。

**信息获取优先级：**

1. **对话历史优先**：如果用户刚完成了一个任务（"把刚才的做成技能"），从对话历史中提取工作流——使用过的工具、执行步骤、用户做的修正、输入输出格式。这种情况下通常不需要额外追问，直接输出你的理解让用户确认即可。
2. **用户描述中提取**：用户说"帮我创建公众号长文的 skill，我是 AI 行业的"，这句话已经包含了技能目标、领域、输出形式，不需要逐条追问。
3. **合理推断**：从技能类型可以推断出大部分细节（输出格式、工作流步骤等），把你的推断整理好，一次性让用户确认或调整。

**只问用户需求**：聚焦于"这个技能要帮你做什么"这类用户能直接回答的问题。触发短语、description 怎么写——这些是平台和你的工作，不要让用户操心。

**一轮对齐**：最多追问用户一轮。将你对技能的理解整理为简洁摘要，附上少量需要用户决定的选项（如果有），一次性发出。用户确认后直接进入 Phase 2。

如果需要调研（搜索文档、查找类似实现、了解最佳实践），先调研再向用户输出你的理解，带着充分上下文来减少对用户的打扰。

### Phase 2：用例到资源规划

在动手写之前，先将 Phase 1 确认的用例逐个拆解，想清楚技能需要哪些附属资源。分析每个用例时问自己：

1. 这个任务如果不靠技能，从零执行需要什么步骤？
2. 哪些步骤每次都会重复（重写代码、翻文档、找模板）？
3. 把它们固化为 scripts / references / assets。

**示例：PDF 编辑技能**

| 用例 | 分析 | 资源 |
|------|------|------|
| "帮我旋转这个 PDF" | 每次旋转 PDF 都要重写同一段代码 | → `scripts/rotate_pdf.py` |
| "提取所有页面为图片" | 同样，重复代码 | → `scripts/pdf_to_images.py` |
| "PDF 里有什么字体？" | 每次要查 PDF 字体属性的 API 用法 | → `references/pdf-api.md` |
| "用公司模板生成报告" | 每次要复制相同的模板文件 | → `assets/report-template.pdf` |

**常见映射关系：**

- 确定性重复代码 → `scripts/`（可直接执行，不占上下文）
- 领域知识、表结构、API 文档 → `references/`（按需加载）
- 模板、样板代码、品牌素材 → `assets/`（用于产出物，不加载到上下文）

规划完成后将资源列表整理好，用户确认后进入 Phase 3。

### Phase 3：编写技能

先用 init_skill.py 初始化目录骨架，再编辑生成的 SKILL.md：

```bash
python3 ~/.workbuddy/skills/skill-creator/scripts/init_skill.py {name}
```

脚本会生成 SKILL.md 模板和 scripts/references/assets 三个资源目录（含示例文件），根据需要将示例文件替换为真实内容或删除，删除不需要的空资源目录。

#### SKILL.md 组件

- **name**：技能标识符，kebab-case，≤64 字符。从用户描述提取核心词转换——"合同审查" → `contract-review`，"周报生成" → `weekly-report`
- **description**：采用统一格式：`<核心能力>。触发：<关键词>。不触发：<排除场景>。` 关键词 3-5 个，覆盖用户常见表达；不触发写绝对排除项。例如："合同审查与风险分析。触发：审合同、检查合同、合同风险评估、上传了合同文件。不触发：合同起草、合同翻译。"
- **agent_created**：YAML frontmatter 中必须包含 `agent_created: true`，确保 SkillManage 工具可以后续修改或删除该技能。缺少此标记的技能将无法通过 SkillManage 进行管理。
- **正文**：技能的具体指示。读取 reference 文件获取编写模式：
  - `references/workflows.md` — 流程组织模式（顺序流程、条件流程、并行流程）
  - `references/output-patterns.md` — 输出格式定义（严格模板 vs 灵活指引、示例模式）

#### 技能结构

```
skill-name/
├── SKILL.md（必填）
│   ├── YAML frontmatter（name + description，必填）
│   └── Markdown 指示正文
└── 附属资源（可选）
    ├── scripts/    - 可执行脚本（确定性/重复性任务）
    ├── references/ - 按需加载的参考文档
    └── assets/     - 输出中使用的文件（模板、图标等）
```

渐进式披露——技能使用三级加载：
1. **元数据**（name + description）—— 始终在上下文中
2. **SKILL.md 正文** —— 技能触发时加载（建议 < 400 行）
3. **附属资源** —— 按需加载（脚本可直接执行无需加载）

SKILL.md 接近上限时，将参考资料拆入 `references/` 并在正文中说明何时读取。

#### 信息架构规则

- **一层引用**：所有附属文件由 SKILL.md 直接链接，不要出现 A 引用 B、B 引用 C 的嵌套链
- **< 400 行**：SKILL.md 超限内容拆分到 `references/`

#### API Key 管理约定

创建需要 API Key 的技能时，遵守以下存储约定：

- **主控目录**：`~/.config/{service}/api_key`
- **编码规则**：敏感 Key 用 Base64 编码存储；客户端级 Key 可明文
- **桥接层**：`.openclaw/.env` 仅作为运行时环境变量桥接，不视为主存储

#### 编写风格

向模型解释事情为什么重要，而不是堆砌生硬的"必须"。让技能具有通用性而非局限于特定示例。

#### 避免的做法

1. **重复 LLM 已有能力** — 技能应提供结构化流程或领域知识，而非"请用优美的语言写作"
2. **引用不存在的工具** — 编写前检查自身工具列表
3. **只说 MUST 不说 WHY** — 解释原因让 Agent 能在边界情况自主判断
4. **超长不拆分** — 超过 400 行的内容拆入 `references/`
5. **没有工作流示例** — 至少 2 个覆盖典型场景的示例
6. **缺少确认门** — 长流程倾向于有用户确认点

#### 质量自查

生成文件后逐项自查（发现问题直接修复，不需要展示给用户）：

| 检查项 | 标准 |
|--------|------|
| frontmatter | name 为 kebab-case ≤64 字符；description 含触发条件和不适用边界 |
| TODO 残留 | SKILL.md 中无 `[TODO` 占位符残留（init_skill.py 模板含大量 TODO，编写完成后应全部替换） |
| 流程清晰度 | 只读各步骤标题，能串成通顺的操作链 |
| 输出格式 | 明确定义了产出物的格式和存放位置 |
| 行数控制 | SKILL.md ≤ 400 行 |

### Phase 4：注册交付

将技能写入对应目录即可完成注册：

- **用户级技能**（默认）：写入 `~/.workbuddy/skills/{name}/`
- **项目级技能**：写入 `{workspace}/.workbuddy/skills/{name}/`

注册完成后告知用户：

> 技能已就绪，当前对话即刻生效。如果以后想修改，可以说"修改我的 XX 技能"。

### Phase 5：打包分发（可选）

如需将技能分享给他人，使用 `package_skill.py` 打包为 `.zip` 文件：

```bash
python3 ~/.workbuddy/skills/skill-creator/scripts/package_skill.py ~/.workbuddy/skills/{name}
# 可选指定输出目录
python3 ~/.workbuddy/skills/skill-creator/scripts/package_skill.py ~/.workbuddy/skills/{name} ./dist
```

脚本会自动完成两步：

1. **校验** — YAML frontmatter 格式、必填字段、命名规范、description 质量、目录结构完整性。不通过直接报错，拒绝打包
2. **打包** — 校验通过后生成 `<skill-name>.zip`，保持原始目录结构，可直接分发安装

通过打包校验的技能即代表质量合格，可以放心分享。

## 二、修改已有技能

当用户说"修改我的 XX 技能"时：

1. **定位**：用 Glob 搜索 `~/.workbuddy/skills/*/SKILL.md`，通过 name 或 description 匹配目标技能
2. **读取并理解**：读取现有 SKILL.md 和相关 references，理解当前结构
3. **修改**：小变动推荐用 Edit 工具精确修改，避免用 Write 重写未变动部分
4. **重新生效**：文件写入后即刻在当前对话生效，无需额外注册步骤

### 编辑市场安装的技能

当用户编辑从市场安装的技能时，修改完成后需要额外标记以防自动更新覆盖：

1. 在技能目录下找到 `_skillhub_meta.json`（或 `_knot_meta.json`）文件
2. 读取该 JSON 文件，设置 `"userModified": true`，写回
3. 如果两个 meta 文件都不存在，说明该技能为本地创建，跳过此步骤，不做任何操作

标记后告知用户：修改已保存，后续市场更新将询问后再覆盖你的改动。

> 注意：仅在文件实际写入成功后再标记，不要在编辑保存前或新建技能时标记。

### 持续迭代

技能不是一次写完就结束的。在真实任务中使用后，往往能发现改进空间：

```
使用技能做真实任务
       ↓
发现卡壳/低效/出错的地方
       ↓
确定 SKILL.md 或附属资源怎么改
       ↓
实施修改（Edit 优先）
       ↓
再测试
```

**建议：** 迭代在刚用完技能之后做效果最好，因为上下文最新鲜，对"哪里不好用"印象最深。拖到第二天细节可能就忘了。
